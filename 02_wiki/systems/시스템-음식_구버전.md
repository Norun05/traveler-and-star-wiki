# 요리 시스템 통합 정리 (문서 저장용)

## 0) 한 줄 개요

**4슬롯(2×2) 재료를 조리도구에 넣으면, 레시피가 있으면 정식 요리로, 없으면 도구가 강제하는 형태로 ‘대체 결과’가 나오며, 도감은 “원종 1칸”만 수록하고 변형은 엔트리 내부 라벨로 처리한다.**

---

## 1) 시스템 철학/목표 (고정)

### 1.1 실행 보장

- 어떤 조리도구/조합이든 **실행은 항상 가능**.
    
- 단, 결과가 항상 “정상 요리”인 건 아니고, **정상/기묘/망함** 같은 등급으로 품질이 갈림.
    

### 1.2 도구 주도(형태 강제)

- 레시피가 없더라도 도구는 결과의 “형태(주스/수프/스무디/구움 등)”를 강제한다.
    
- 즉 “레시피=결과를 특정”이고, “도구=결과의 형태를 특정”임.
    

### 1.3 도감은 ‘조합 나열’이 아니라 ‘조건(범위)’ 중심

- 같은 원종 요리로 가는 조합이 여러 개인 건 정상.
    
- 도감은 조합 목록이 아니라 **원종 1개 엔트리 + 조건(범위) + 변형 라벨 + 내 조합 로그**로 관리.
    

---

## 2) UI 구조 (고정)

- 상단: **미리보기 이미지(a)**
    
- 중앙: **조리도구/조리 버튼/결과 영역**
    
- 하단: **2×2 재료 슬롯(4칸)**
    
- 오른쪽: **재료 인벤(냉장고/보관함)**
    
- 레시피/도감 패널: **토글(켜기/끄기)**
    

---

## 3) 재료 데이터 모델 (고정)

### 3.1 “카테고리”와 “태그/수치”를 분리

- 카테고리(인벤 정렬/검색용): 플레이 동선 기준(예: 베이스/메인/향신/냉각 등)
    
- 태그/수치(레시피 판정용): 맛/물성/가공가능성/효과
    

### 3.2 재료가 가져야 하는 최소 구성

- 분류 태그: `base`, `main`, `topping`, `spice`, `cooling` 등(너가 정함)
    
- 성질 수치: `sweet`, `sour`, `water`, `fat`, `protein`, `cold`, `hot` 등
    
- 기능 태그: `blendable`, `juiceable`, `sauceable`, `boilable` 등
    
- 선택: `edible_raw`(인벤 생식 가능 여부), `prep_bonus`(손질 보너스)
    

### 3.3 “단일 재료 조리”를 살리는 원칙

- 특정 성분이 0이면 시스템이 막히니까, 필요 시 재료에 **미량 수치**를 넣어 “최소 성립”을 보장한다.
    
- 이건 현실 고증이 아니라 **플레이/판정 안정성** 목적.
    

---

## 4) 조리도구(스테이션) 규칙 (고정)

도구는 결과의 “형태”를 강제한다.

- 믹서기: 스무디/퓨레/슬러시(갈림 계열)
    
- 주서기: 주스/착즙/펄프 분리 계열
    
- 쉐이커: 혼합/차게 만들기(얼음은 coolant 처리하기 쉬움)
    
- 냄비/가열: 수프/탕/시럽/졸임 계열
    
- 기타 “말도 안 되는 도구”도 실행은 되며, 해당 도구의 “행위”에 맞춰 형태만 정의하면 됨.
    

---

## 5) 판정 파이프라인 (고정)

조리 버튼을 누르면 아래 순서로만 판정한다. (이 순서 안 지키면 시스템이 바로 산으로 감)

1. **명시 레시피 매칭(우선)**
    
    - 정확 조합 또는 태그/범위 조건으로 특정 요리를 확정
        
2. **도구 기반 형태 변환(레시피 없을 때)**
    
    - “미확정 ○○(스무디/수프/주스…)” 같은 대체 결과 생성
        
3. **변형(라벨) 계산**
    
    - 온도/농도/단맛/점도/비율(소스/면 등)로 라벨 부여
        
    - 예: `차가운`, `무가당`, `진한`, `소스 부족` 등
        
4. **품질 등급 결정(항상 산출)**
    
    - `OK`: 정상
        
    - `WEIRD`: 기묘(약한 페널티/특수효과/도감용 재미)
        
    - `RUIN`: 망함(먹을 수는 있으나 효율 최악, 혹은 만족도 크게 하락)
        

---

## 6) 얼음 처리 규칙 (고정)

“숫자만”으로 하지 말고 우선순위를 둔다.

### 6.1 우선순위

1. **도구 우선**: 주서기/쉐이커/냄비는 기본적으로 “얼음 갈림”이 아님
    
2. **재료 역할 태그**: `ice:blend` / `ice:garnish` / `ice:coolant`
    
3. **폴백 규칙**(레시피/역할 없을 때)
    

### 6.2 폴백 예시(믹서기 기준)

- 얼음 0: 일반(주스/퓨레)
    
- 얼음 1: 기본은 **동동(가니시)**
    
- 얼음 2+: **갈림(슬러시)**
    
- 선택 예외: 유제품이 있으면 얼음 1도 갈림으로 보내도 납득 쉬움
    

### 6.3 “4+1(가니시)” 표현

- UI 슬롯은 4칸 고정 유지.
    
- 얼음이 `garnish`로 판정되면 **결과 이미지 레이어로만** “컵 위 얼음”을 남긴다(데이터는 4개 소모, 표현은 동동).
    

---

## 7) 1개 재료 처리: 인벤 섭취 vs 주방 손질 (고정)

### 7.1 인벤에서 “먹기(Consume)”

- `edible_raw`면 인벤에서 즉시 섭취 가능.
    
- 효과는 보통 **소량**(편의성 대신 효율 낮음).
    

### 7.2 주방에서 “손질(Prep)”

- 1개로도 가능하게 하되, 결과는 완제품 요리가 아니라 **가공품/중간재**가 원칙.
    
- 가공품은
    
    - 이후 요리에서 품질/안정성 보너스
        
    - 혹은 재료 분할/압축(슬롯 절약)을 제공
        

### 7.3 가공품의 정의(확정)

- **가공품은 ‘레시피 필수템’이 아니라 ‘슬롯을 사는 재료’**다.
    
- 그래서 가공품이 있으면 토핑/향신/치즈 같은 변형 재료를 꽂을 칸이 생기고, 그게 시스템 깊이가 됨.
    

---

## 8) 레시피 도감 구조 (고정)

### 8.1 리스트 화면

- 도감 리스트(번호/칸)에는 **원종만** 표시.
    
- 변형은 리스트에 별도 항목으로 늘리지 않음(번호 폭발 방지).
    

### 8.2 엔트리 내부(원종 1개 안에서 처리)

엔트리 1개는 3층 구조로 고정:

A) **기본 정보(고정)**

- 원종 이름/대표 이미지/설명
    

B) **조건(범위) + 변형 라벨 규칙**

- 예: “○○ 계열 ≥ N”, “도구 타입”, “온도/농도/비율 라벨” 등
    

C) **내가 만든 조합 로그**

- 최근 3개/즐겨찾기 3~5개 등(너가 UI 용량 정함)
    
- 로그는 “조합 나열”이 아니라 **리플레이 편의**용
    

---

## 9) 미리보기/미발견 UX (고정)

레시피 미발견 상태에서 정보 제공은 이렇게 간다.

- a(이미지): `실루엣` 또는 `블러`(정식 이미지 금지)
    
- b(이름칸): `???`만 띄우지 말고 **서술형 자동조합**
    
    - 예: `차가운 · 산미 · 음료`, `따뜻한 · 크리미 · 수프`
        
- 완성/발견 순간:
    
    - 정식 이름/정식 이미지 공개 + 도감 등록
        

---

## 10) “말도 안 되는 조리도구도 실행” 처리 (고정)

- 실행 불가를 두지 않는다.
    
- 대신 “불가능 요소”는 삭제가 아니라 **부작용/품질 하락**으로 반영한다.
    
    - 예: 냄비에 얼음 → 묽어짐/향 약화/품질↓ 같은 식으로 결과가 “이상해진 이유”를 납득 가능하게 만든다.
        

---

## 11) 변형 라벨 예시 세트 (고정)

원종 엔트리 내부에서만 쓰는 라벨(카테고리화) 예시:

- 온도: `차가운 / 미지근한 / 따뜻한 / 뜨거운`
    
- 단맛: `무가당 / 가당 / 진한 단맛`
    
- 농도/점도: `묽은 / 표준 / 진한`
    
- 비율(예: 메인+소스): `정상 코팅 / 소스 부족 / 소스 과다`
    
- 품질: `OK / WEIRD / RUIN`

[요약]

1. 요리 시스템은 **4슬롯(2×2) + 도구가 형태 강제 + 실행 보장**으로 고정한다.
    
2. 재료는 **카테고리(정렬)**와 **태그/수치(판정)**를 분리하고, 필요 시 미량 수치로 “단일 재료 성립”을 보장한다.
    
3. 판정은 **레시피 매칭 → 도구 기반 대체 결과 → 변형 라벨 → 품질 등급(OK/WEIRD/RUIN)** 순서로 고정한다.
    
4. 얼음은 **도구 우선 → 역할 태그 → 폴백**으로 갈림/동동/제거를 결정하고, “4+1 가니시”는 **표현 레이어**로 처리한다.
    
5. 도감은 **원종만 리스트에 1칸**, 변형은 엔트리 내부 **라벨+조건+내 조합 로그**로 처리해 엔트리 폭발을 막는다.
